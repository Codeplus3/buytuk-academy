## فهم المشتري

### أنواع المشترين

```
نوع 1 — مدير المدرسة الرؤيوي
  الشخصية: متحمس للتقنية، يؤمن بالبيانات، يريد التميز
  الألم   : "لا أعرف ما يجري في فصولي فعلاً"
  الرسالة : "سترى صورة مدرستك كاملة لأول مرة — بلغة الأرقام"
  القرار  : سريع — يحتاج Demo جيداً وسعراً منطقياً

نوع 2 — مدير المدرسة الحذر
  الشخصية: يهتم بالتغيير لكن يخشى المخاطرة
  الألم   : "المعلمون مشغولون — لا وقت لأداة جديدة"
  الرسالة : "5 دقائق لتسجيل الجلسة — ثم النظام يعمل وحده"
  القرار  : يحتاج Pilot صغير + قصة نجاح مشابهة

نوع 3 — مشتري مؤسسي (إدارة تعليمية / وزارة)
  الشخصية: يعمل بـ RFP و Procurement رسمي
  الألم   : "لا معيار موحد لقياس جودة التعليم عبر المدارس"
  الرسالة : "BuyTuk يُعطيك لوحة بيانات موحدة لكل مدارسك"
  القرار  : بطيء (6-18 شهر) — يحتاج Legal + Security + References
```

## مراحل دورة المبيعات

### المرحلة 1: الاكتشاف (Discovery)
```
الهدف: افهم مشكلتهم قبل أن تعرض حلك

الأسئلة الذهبية:
  "كيف تعرف حالياً أن طالباً يحتاج تدخلاً؟"
  "كم وقت يقضي معلموك في تقييم الطلاب أسبوعياً؟"
  "ماذا يحدث عندما تكتشف فجوة في تعلم طالب؟"
  "هل لديك بيانات موضوعية لتقديمها للوزارة أو لأولياء الأمور؟"
  "ما الذي جربتموه قبل للحل؟ وماذا نجح / لم ينجح؟"

ما تبحث عنه:
  ✅ يصف مشكلة حقيقية بحرارة (ليس فقط "نعم مشكلة")
  ✅ يملك صلاحية القرار أو يصل لمن يملكها
  ✅ لديه ميزانية (حتى محدودة)
  ✅ التوقيت مناسب (لا أزمات حالية)
```

### المرحلة 2: العرض التجريبي (Demo)
```
الهدف: يخرج المدير يتخيل كيف سيحل BuyTuk مشكلته — بالتحديد

بنية Demo الـ 30 دقيقة:
  5 دقائق  — "قبل" المشهد: "الآن لما يكتشف المعلم مشكلة — ماذا يفعل؟"
  10 دقائق — تسجيل جلسة + عرض النتيجة (Live — لا شرائح)
  5 دقائق  — الفجوة تُكتشَف تلقائياً + بروتوكول العلاج
  5 دقائق  — لوحة المدير: صورة المدرسة كاملة
  5 دقائق  — "بعد" المشهد: "تخيّل بعد 3 أشهر..."

قواعد Demo:
  ✅ استخدم بيانات تجريبية تشبه واقعهم (مدارسة، مواد مماثلة)
  ✅ اترك المدير يضغط ويتجول (لا تكن أنت من يضغط)
  ✅ إذا سألوا سؤالاً ما تعرف إجابته → "سأتحقق وأرد خلال 24 ساعة"
  ✅ في آخر 2 دقيقة: "ما الذي أعجبك أكثر؟" + "ما الذي تريد رؤيته أكثر؟"
  ✗ لا تعرض جميع الميزات — الكثير يُشتّت
  ✗ لا تتحدث عن المنافسين إلا إذا سُئلت
```

### المرحلة 3: الاعتراضات (Objections)

```
"المعلمون مشغولون جداً"
→ "نفهم — لذلك صمّمنا الجلسة لتكتمل في 5 دقائق.
   في الـ Pilot نقيس: هل يستغرق المعلمون أكثر من 5 دقائق؟
   إذا نعم — نُعدّل."

"لدينا بالفعل نظام"
→ "ما النظام؟ نحن لسنا بديلاً — نحن طبقة التشخيص فوقه.
   [اسم النظام] يُدير الحضور والدرجات — نحن نكتشف لماذا يتأخر الطالب."

"السعر مرتفع"
→ "دعنا نحسب: إذا حسّنت أداء 10 طلاب في خطر الرسوب —
   ما قيمة ذلك لسمعة مدرستك + رضا أولياء الأمور؟
   بالمقابل، Pilot لمدة 12 أسبوعاً بـ[السعر الرمزي]."

"نريد التفكير"
→ "بالطبع — ما الذي تحتاج التفكير فيه؟"
   (افهم الاعتراض الحقيقي — لا تقل "خذ وقتك")
   "سأرسل لك ملخصاً + حالة نجاح مشابهة + عرض السعر
   ونتحدث الأسبوع القادم؟ متى يناسبك؟"

"نحتاج موافقة إدارة أعلى"
→ "من يجب أن يشاهد هذا معك؟ يمكنني أن أقدم Demo قصيراً لهم"
   (لا تترك الشخص الآخر يبيع بدلاً منك)
```

### المرحلة 4: الإغلاق (Closing)
```
الهدف: التزام واضح — ليس "سنتحدث لاحقاً"

تقنيات الإغلاق:
  الـ Pilot Close:
  "بدلاً من القرار النهائي الآن — ما رأيك أن نبدأ 12 أسبوعاً بـ[سعر رمزي]؟
   إذا لم يُثبت قيمته — أنت لا تخسر شيئاً مهماً."

  الـ Next Step Close:
  "الخطوة التالية هي: أنت تُرسل قائمة المعلمين والطلاب
   وأنا أُعدّ النظام خلال 3 أيام.
   متى تستطيع إرسال القائمة؟"

  الـ Deadline Close (بحذر — لا تختلق):
  إذا كان هناك عرض سعر موقوت حقيقي: استخدمه
  إذا لم يكن → لا تختلق ضغطاً وهمياً (يُقلل الثقة)

علامات جاهزية الشراء:
  ✅ يسأل عن السعر التفصيلي (ليس فضولاً عاماً)
  ✅ يتساءل "ماذا يحتاج من معلمينا؟" (تفكير تنفيذي)
  ✅ يطلب مشاركة Demo مع شخص آخر في المدرسة
  ✅ يسأل "متى يمكن أن نبدأ؟"
```

## القواعد الذهبية للمبيعات في BuyTuk

```
1. نبيع الأثر — لا الميزات
   ❌ "لدينا 6 محركات ذكاء"
   ✅ "ستعرف لأول مرة لماذا يتأخر طالبك — وماذا تفعل بالتحديد"

2. الصدق قبل البيع
   إذا المنتج لا يحل مشكلتهم → قلها
   العميل الخاطئ يُكلّف أكثر من العميل المفقود

3. السرعة في المتابعة
   كل Demo بدون متابعة خلال 24 ساعة = Demo ضائع
   أرسل الملخص + عرض السعر في نفس اليوم

4. سجّل كل شيء
   كل محادثة → في CRM (حتى "قال لا")
   الـ Pattern هو الذي يُحسّن المبيعات

5. لا تعد بما لا يستطيع المنتج تقديمه
   راجع SYSTEM_BOUNDARIES.md قبل أي وعد
   الوعد المُخلَف = إلغاء اشتراك + سمعة سيئة
```

## مؤشرات المبيعات

```
نتابع أسبوعياً:
  Demos مُنجَزة
  Proposals مُرسَلة
  Pilots مُوقَّعة
  Conversions (Pilot → Paid)

نتابع شهرياً:
  MRR الجديد
  Win Rate (% من Demos → Paid)
  Average Sales Cycle (أيام من Demo → Sign)
  Average Deal Size

الأهداف الأولى (بعد Pilot):
  Win Rate > 25%
  Sales Cycle < 30 يوم
```
