## المشاكل الأكثر شيوعاً

---

### 🔴 المشكلة 1: "لا أستطيع تسجيل الدخول"

```
الخطوة 1: اسأل المستخدم
  - ما رسالة الخطأ بالضبط؟
  - أي متصفح يستخدم؟
  - هل نجح الدخول من قبل؟

الخطوة 2: تحقق من الحساب
  □ ابحث عن البريد في قاعدة البيانات
  □ هل الحساب موجود؟
  □ هل الحساب مُفعَّل (is_active = true)؟
  □ هل الدور (role) صحيح؟

الخطوة 3: تحقق من الباقة
  □ هل المدرسة (tenant) نشطة؟
  □ هل منتهية صلاحية الاشتراك؟

الخطوة 4: الحلول
  - كلمة المرور خاطئة → أرسل "إعادة تعيين كلمة المرور"
  - الحساب غير موجود → تحقق من هجاء البريد
  - الحساب غير مُفعَّل → فعِّله من لوحة Admin
  - Tenant غير نشط → تواصل مع فريق المبيعات

وقت الحل المستهدف: < 10 دقائق
```

---

### 🔴 المشكلة 2: "بياناتي اختفت"

```
الخطوة 1: اسأل المستخدم
  - ما نوع البيانات التي اختفت؟ (جلسات، طلاب، نتائج)
  - متى كانت موجودة آخر مرة؟
  - هل يستخدم نفس الحساب والمتصفح؟

الخطوة 2: تحقق من قاعدة البيانات
  □ هل البيانات موجودة في DB؟
     SELECT * FROM [table] WHERE student_id = ? AND tenant_id = ?
  □ إذا موجودة → مشكلة في الواجهة (ليس في البيانات)
  □ إذا غير موجودة → احتمالات:
     - كان يستخدم localStorage القديم (قبل v3.0)
     - تم استخدام حساب مختلف
     - تم الحذف عن طريق الخطأ (راجع Audit Log)

الخطوة 3: راجع Audit Log
  SELECT * FROM audit_logs
  WHERE tenant_id = ? AND action = 'DELETE'
  ORDER BY created_at DESC LIMIT 100;

الخطوة 4: إذا كانت البيانات محذوفة عن طريق الخطأ
  □ راجع آخر Backup
  □ استعد البيانات المفقودة فقط
  □ وثّق الحادثة

وقت الحل المستهدف: < 1 ساعة
```

### 🟡 المشكلة 3: "المنصة بطيئة جداً"

```
الخطوة 1: اسأل المستخدم
  - أي صفحة بطيئة؟
  - منذ متى وهي بطيئة؟
  - هل الإنترنت لديه بطيء عموماً؟

الخطوة 2: تحقق من جهتنا
  □ راجع OBSERVABILITY.md → API latency metrics
  □ هل المشكلة عامة أم للمستخدم فقط؟
  □ تحقق من: GET /api/health/deep

الخطوة 3: إذا المشكلة عامة
  → راجع OBSERVABILITY.md → Runbook "عند بطء الـ API"

الخطوة 4: إذا المشكلة للمستخدم فقط
  □ اطلب منه تجربة متصفح آخر
  □ اطلب منه clear cache
  □ اطلب منه قياس سرعة الإنترنت
  □ إذا الإنترنت بطيء → مشكلة في شبكة المدرسة (خارج نطاقنا)

وقت الحل المستهدف: < 30 دقيقة
```

---

### 🟡 المشكلة 4: "النتيجة المحسوبة تبدو خاطئة"

```
الخطوة 1: اسأل المستخدم
  - ما النتيجة التي يرى؟
  - ما النتيجة التي يتوقع؟
  - ما المعيار والمحرك؟

الخطوة 2: راجع منطق الحساب
  □ افتح DECISION_LOGIC.md → قسم المحرك المعني
  □ احسب يدوياً بنفس الصيغة
  □ قارن مع ما حسبه النظام

الخطوة 3: تحقق من البيانات المُدخَلة
  □ SELECT * FROM [session_table] WHERE id = ?;
  □ هل الدرجات المُدخَلة صحيحة؟
  □ هل الحساب تم بالصيغة الصحيحة؟

الخطوة 4: إذا كان الحساب خاطئ فعلاً
  □ وثّق الـ Bug في TECH_DEBT_REGISTER.md أو GitHub Issues
  □ لا تُعدّل النتيجة يدوياً في DB بدون ADR
  □ أخبر المستخدم بالجدول الزمني للإصلاح

ملاحظة: إذا كان المستخدم "يتوقع" نتيجة مختلفة بسبب رأيه الشخصي
  → اشرح له DECISION_LOGIC.md ومنطق الحساب
  → لا تغيّر الخوارزمية لطلب فردي
```

### 🟡 المشكلة 5: "لا يمكنني إضافة طالب جديد"

```
الخطوة 1: اسأل المستخدم
  - ما رسالة الخطأ؟
  - هل جرّب تحديث الصفحة؟

الخطوة 2: تحقق من الصلاحيات
  □ هل المستخدم لديه دور Teacher أو Admin؟
  □ هل الـ Class موجودة ومرتبطة بـ Teacher هذا؟

الخطوة 3: تحقق من حدود الخطة
  □ هل المدرسة وصلت للحد الأقصى من الطلاب في خطتها؟

الخطوة 4: تحقق من API
  □ راجع /api/logs → POST /api/users → ما رمز الخطأ؟
  □ إذا 403 → مشكلة صلاحيات
  □ إذا 400 → بيانات غير صحيحة (validation error)
  □ إذا 500 → خطأ في الخادم ← يحتاج تحقيق تقني
```

## مصفوفة التصعيد

```
المستوى 1 — Support Agent (دعم عادي)
  → مشاكل تسجيل الدخول
  → مشاكل إضافة مستخدمين/بيانات
  → أسئلة عن كيفية الاستخدام
  → وقت الحل: < 1 ساعة

المستوى 2 — Senior Support / Technical
  → بيانات مفقودة يُشتبه في حذفها
  → مشاكل أداء لمدرسة بعينها
  → نتائج تبدو خاطئة
  → وقت الحل: < 4 ساعات

المستوى 3 — Engineering Team
  → أخطاء في الخادم (5xx)
  → بيانات مفقودة بعد تحقيق مستوى 2
  → ثغرات أمنية مشتبه بها
  → وقت الحل: < 24 ساعة

المستوى 4 — Incident Command
  → توقف كامل للخدمة
  → اختراق أمني مؤكد
  → فقدان بيانات واسع النطاق
  → وقت الاستجابة: < 1 ساعة
```

