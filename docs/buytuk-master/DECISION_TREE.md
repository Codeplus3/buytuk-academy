## 🌳 شجرة 1: أريد إضافة محرك ذكاء جديد

```
هل المحرك الجديد يقيس معياراً تعليمياً موثَّقاً؟
├── لا → أعد تعريف ما تريد قياسه أولاً
└── نعم ↓

هل يتبع النمط الطبقي: Calculator → Service → Routes → Panel؟
├── لا → ادرس محرك Reading أو Writing أولاً
└── نعم ↓

هل يحتاج جداول جديدة في DB؟
├── نعم → أضف الجدول الجديد فقط (لا تُعدّل الجداول الحالية)
└── لا ↓

هل يحتاج API Routes جديدة؟
├── نعم → أضف routes جديدة في api-server
└── لا ↓

هل يحتاج تغيير في OpenAPI spec؟
├── نعم → حدّث lib/api-spec/openapi.yaml ثم شغّل codegen
└── لا ↓

الخطوات:
1. lib/db/src/schema/ → جدول جديد
2. lib/api-spec/openapi.yaml → endpoints جديدة
3. pnpm --filter @workspace/api-spec run codegen
4. artifacts/api-server/src/routes/ → router جديد
5. artifacts/school-platform/src/[engine]/ → Calculator + Service
6. artifacts/school-platform/src/components/ → TeacherPanel + StudentPanel
7. TeacherPortal.tsx + StudentPortal.tsx → سطر إضافة واحد لكل ملف
8. lib/db/src/seeds/seed.ts → بيانات تجريبية
9. HANDOVER.md + DECISION_LOGIC.md → توثيق

لا تُعدّل أي محرك سابق إلا بـ ADR رسمي.
```

## 🌳 شجرة 2: أريد إضافة صفحة/Portal جديدة

```

├── لا → أضف الدور في auth system + RBAC middleware أولاً

├── نعم → إضافة endpoints جديدة (شجرة 3)
└── لا ↓

الخطوات:
1. حدد: ما البيانات التي يحتاجها هذا الـ Portal؟
2. تحقق أن جميع الـ API endpoints موجودة
3. أنشئ ملف [Role]Portal.tsx
4. اربطه في App.tsx بـ route حسب الدور
5. حدّث ARCHITECTURE.md
```

## 🌳 شجرة 3: أريد تعديل قاعدة البيانات

```
نوع التعديل؟

├── [إضافة جدول جديد]
│   → آمن — ابدأ بـ lib/db/src/schema/
│   → شغّل: pnpm --filter @workspace/db run push
│   → لا تنسَ: Index على الحقول الأكثر استخداماً
│
├── [إضافة حقل جديد لجدول موجود]
│   → آمن إذا كان nullable أو له default
│   → أضف الحقل + شغّل push
│   → تحقق من كل query تستخدم هذا الجدول
│
├── [تعديل حقل موجود (اسم أو نوع)]
│   → خطير — يحتاج ADR أولاً
│   → Migration script يجب أن يُكتب
│   → اختبر على نسخة من البيانات أولاً
│
├── [حذف جدول أو حقل]
│   → خطير جداً — يحتاج ADR + موافقة قيادة المشروع
│   → تحقق من جميع الكود الذي يستخدم هذا الجدول/الحقل
│   → Soft delete أفضل من الحذف الفعلي
│
└── [تعديل Schema للـ Multi-Tenancy]
    → يحتاج خطة migration كاملة
    → راجع TD-007 في TECH_DEBT_REGISTER.md
    → لا تفعله بمفردك — قرار فريق
```

## 🌳 شجرة 4: أريد إضافة مكتبة جديدة

```
ما الحاجة التي تحلها هذه المكتبة؟
└── اكتب الإجابة صراحة ↓

هل يمكن تنفيذ الحاجة بكود بسيط بدون مكتبة (< 50 سطر)؟
├── نعم → لا تُضف مكتبة — اكتب الكود
└── لا ↓

تحقق من هذه المعايير:
├── هل آخر تحديث للمكتبة: < 6 أشهر؟
├── هل عدد النجوم على GitHub: > 1000؟
├── هل عدد المتابعين (weekly downloads): > 10,000؟
├── هل الرخصة مقبولة (MIT/Apache/BSD)؟
└── هل يمكن استبدالها إذا احتجنا لاحقاً؟

إذا مررت كل المعايير:
├── الحزمة للـ Frontend فقط؟ → devDependencies
├── الحزمة للـ Backend (runtime)؟ → dependencies
└── سجّل في DEPENDENCY_POLICY.md لماذا أضفتها
```

## 🌳 شجرة 5: أريد تغيير منطق حساب الدرجات

```
└── لا ↓

└── يحتاج ADR + تحديث standardsConstants.ts
    مع مراعاة أن الجلسات القديمة تبقى بالمعايير القديمة
```

## 🌳 شجرة 6: أريد إصلاح bug

```
هل الـ Bug في بيانات الإنتاج؟
├── نعم → لا تُعدّل البيانات مباشرة بدون:
│         1. نسخة احتياطية أولاً
│         2. script مراجعة قبل التطبيق
│         3. تسجيل في Audit Log
└── لا ↓

هل الـ Bug يؤثر على حساب الدرجات؟
├── نعم → هل نُعيد حساب النتائج القديمة؟
│         → لا تلقائياً — قرار Product Owner
│         → راجع DECISION_LOGIC.md
└── لا ↓

هل الـ Bug في محرك Stable؟
├── نعم → الإصلاح يتبع Sprint Rule (تعديل بالإضافة فقط)
│         وثّق في TECH_DEBT_REGISTER.md إذا كان دَيناً قديماً
└── لا ↓

الإصلاح المعتاد:
1. اكتب test يُثبت وجود الـ Bug
2. صلّح الكود
3. تحقق أن الـ Test يمر
4. تحقق أن الـ E2E Tests الأخرى لا تزال تمر
5. وثّق في Sprint Report
```
